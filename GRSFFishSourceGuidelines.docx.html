GRSF-FishSource Records approvals

UNDP-SFP OIC Project – Arturo Muñoz Albero

November 2022

  1. Introduction

The present document aims to provide a guideline for GRSF Records checks, in order to settle a standardized process to be followed when new records are uploaded to the GRSF from RAM Legacy database actualizations. For this, an understanding of the three databases that constitute the source of GRSF is needed. In this introduction a basic presentation of the databases is provided, but we encourage to explore thoroughly and get familiar with each of them.

  1. RAM Legacy Database (RAMLDB) - https://www.ramlegacy.org/

The RAM Legacy Stock Assessment Database is a compilation of stock assessment results for commercially exploited marine populations from around the world. It is inspired by Dr. Ransom A. Myers’ original stock-recruitment database (http://wayback.archive-it.org/org-990/20170328023224/http://www.mscs.dal.ca/~myers/welcome.html), which is no longer being updated. To learn more about RAMLDB visit https://www.ramlegacy.org/

The RAMLDB is in “R” and Excel format, and can be downloaded at https://zenodo.org/record/4824192#.Y1D58nZByUk

  1. Fishery and Resources Monitoring System (FIRMS) - http://firms.fao.org/firms/en

FIRMS system is part of the Fisheries Global Information System (FIGIS) of FAO. Information provided by FIRMS partners is organized in a database and published in the form of fact sheets. The primary aim of the Fisheries and Resources Monitoring System (FIRMS) is to provide access to a wide range of high-quality information on the global monitoring and management of fishery marine resources. To learn more about FIRMS visit http://firms.fao.org/firms/about/en

  1. FishSource from the Sustainable Fisheries Partnership - https://www.fishsource.org/

FishSource is a publicly available online resource about the status of fisheries, fish stocks, and aquaculture. FishSource compiles and summarizes publicly available scientific and technical information and presents it in an easily interpretable form. To learn more about FishSource visit https://www.fishsource.org/about

  1. Global Record of Stocks and Fisheries (GRSF) - https://i-marine.d4science.org/group/grsf

The GRSF integrates data from the three described authoritative sources: FIRMS (Fisheries and Resources Monitoring System), RAM (RAM Legacy Stock Assessment Database) and FishSource (Program of the Sustainable Fisheries Partnership). To learn more about GRSF visit https://i-marine.d4science.org/web/grsf/data-catalogue

  1. Identifying new pending records

When an update including new records is implemented in RAMLDB, the GRSF will highlight each new record in “pending status”. This means that the record needs to be validated and checked for overlaps with the other databases records (FIRMS, FishSource). In this chapter we will go through the process of identifying the new pending records and setting the workbook for properly report the analysis results.

  1. The GRSF competency queries

In the GRSF, we can extract specific data views from the database in order to obtain selected information about the stock or fisheries records that are available. This extraction is carried through Competency Queries at https://i-marine.d4science.org/web/grsf/competency-queries

For validating and checking overlaps of new RAMLDB records, the Competency Query that we will use is G4.1 - List GRSF Stock records with their original IDs

  1. G4.1 - List GRSF Stock records with their original IDs

This Competency Query provides us a full list of all the stock records available in the GRSF, together with the following valuable information for our records validation process:

  • UUID. Universally Unique Identifier set for each of the GRSF records.
  • GRSF Name. Name of the stock record.
  • GRSF Semantic id. Semantic identifier of the stock record, constituted by the following rule:

“asfis:XXX + aaa:area_code1;bbb:area_code2;…”

As it is possible to note, the semantic identifier contains two separate parts of information separated by the “+”. At the left side, we have the species information, where “XXX” is the ASFIS code of the species of the record. For example for Atlantic bluefin tuna (Thunnus thynnus), the ASFIS code is “BFT”, so the left side of the Semantic identifier would be “asfis:BFT”.

At the right side of the “+”, the record area is defined by a combination of official areas, with the following fields:

  • “aaa”, “bbb” are the namespacers that define an area code system
  • “area_codeN” is the area code, according to the area code system

For example, for FAO Major Fishing Areas, the namespacer is “fao”. If we want to refer to FAO Major Fishing Areas 21 and 27, the right side of the semantic identifier would be: “fao:21;fao:27”

Therefore, the semantic identifier stock record of Atlantic bluefin tuna in FAO Major Fishing Areas 21 and 27 would be: “asfis:BFT + fao:21;fao:27”.

  • Short name. Short name of the stock record
  • Type. Type of record, depending on if it is an assessment unit or a marine resource.
  • Status. Approved if it has been checked or Pending if the record needs to be validated.
  • FIRMS code. Code of the record in the FIRMS database, if applicable.
  • RAM code. RAM stockid code of the record in RAMLDB, if applicable.
  • FishSource code. FishSource ID code in the FishSource database, if applicable.

  1. Setting the workbook

In order to work in a standardized and organized format, the following fields are advised to be added in the Excel file in which we will work for validating the new RAMLDB records and check for overlaps with other FIRMS or FishSource records.

  • Action for FishSource. To note actions that should be followed by FishSource specialists. These actions can be clarifications, errors identified in the FishSource database that need to be amended or other suggestions.
  • Action for RAM. To note actions that should be followed by RAMLDB specialists. These actions can be clarifications, errors identified in the RAMDLB database that need to be amended or other suggestions.
  • Action for FIRMS. To note actions that should be followed by FIRMS specialists. These actions can be clarifications, errors identified in the FIRMS database that need to be amended or other suggestions.
  • FORTH Action. Action to be implemented in the GRSF. There are four standardized possible actions to be implemented: APPROVE/MERGE/ARCHIVE/NO ACTION. For further information about each action, please go to 3.1.FORTH Actions.
  • FORTH Dominant record. It stablishes which record will be maintained as dominant when a merge is implemented. The current identification for the dominant record is the one that better describes the record area or the record name.
  • Comments. Private notes for keeping track of work done, note doubts, etc.

  1. Action identification

  1.  FORTH Action

For ease of implementation, the actions have been standardized in APPROVE/MERGE/ARCHIVE or NO ACTION. Below each action is explained and field filling instructions are given.

  • APPROVE – When the record is unique and has been validated and checked without any overlap identified from other database records. The FORTH Action field will be filed with “APPROVE”.

  • MERGE + APPROVE – When the record has been validated and checked, finding an overlap with a FIRMS or FishSource record. The FORTH Action field will be filed with:

“MERGE + APPROVE RAMLDB record “XXXXX” (UUID xxxxxxx) with FIRMS record ID “YYYYY” (UUID yyyyyyyyyy) and FishSource record ID “ZZZZ” (UUID zzzzzzzzz)”

  • XXXXX is the RAMLDB stockid (ram_code)
  • xxxxxxx is the UUID of the RMALDB record at the GRSF
  • YYYYY is the FIRMS ID (firms_code)
  • yyyyyyy is the UUID of the FIRMS record at the GRSF
  • ZZZZ is the FishSource ID (fishsource_code)
  • zzzzzzzz is the UUID of the FishSource record at the GRSF

  • ARCHIVE – When a record is no longer applicable because it is split by the source in several records or is no longer assessed. The FORTH Action will be “ARCHIVE”.

  • NO ACTION – When there is an Action for FishSource/RAMLDB/FIRMS that needs to be implemented before the record can be approved/merged or archived. The FORTH Action will be “NO ACTION. Waiting for FishSource/RAMLDB/FIRMS to follow up” depending on if the action identified is for FishSource, RAMLDB or FIRMS.

  1. Overlapping records check

In order to identify the FORTH action needed for each stock, there are a number of steps that can always be applied. Not all cases can me analyzed in the same way, as each case is different, but the following standard steps are of great help for identifying potential overlaps or uniqueness of the pending record:

  1. Identify all other records of the same species, both pending and approved ones, from other databases (FIRMS and FishSource). It is important to note that records can not be merged with records from the same database (i.e., not possible to merge a RAMLDB record with another RAMLDB record)
  2. Compare all other records areas with the area of our pending record.
  3. Discard all other records of the same species with completely different or distant areas.
  4. Identify source of the record at the source database (stock assessment document) for all potential merges.

Also in other cases, with less clear overlaps of records, being able to consult the source of each record is key for identifying the right overlaps.

Note that just with this filtering and following the same process, we could also identify the potential merges for two other pending RAMLDB records: LINGNEATL and LINGI-II, as well as the FishSource record ID 2014.

  1.  Verifying overlaps of records

The most robust process for verifying that records can be merged is to check that the source of information is the same. For that, we will need to check in each of the three databases that feed the GRSF and find the source of each record.

  1. RAMLDB Records

The RAMLDB database is available at https://zenodo.org/record/4824192#.Y1D58nZByUk for download.

When we open the Excel file, there are a number of tabs that we should be familiar with. For identifying the source of the record, we will focus in the “assessment” tab. In this tab, we can filter by the stockid (ram_code) of the record that we want to check. Columns “assesssource”, “notes” and “pdffile”, together with “assesscomments”, provide valuable information to identify the source of the RAMLDB record.

In some cases, the stock assessment pdf file is not available, or the source provides a link that is expired or is not working due to permissions not granted. Even in this cases, it is possible to trace the source of the record by the information provided in the indicated columns by research following the information available.

Following the same example of the previous chapter, for the RAMLDB record stockid LINGVb, we filter in the “assessment” tab of the RAMLDB Excel and obtain the list below:
